## legacy 版本错位结论
### 1. 你当前判断基本正确：存在明显“版本错位”
- 当前前端来自 `front0106`，接口契约是 `/getGoodsByCate`、`/createOrder` 这一套。
- 当前后端来自 `Beyeah1211`，主能力是另一套路由（`/search`、`/shop-cart`、`/saveOrder`、`/orders`），并且偏服务端页面模式。

### 2. 并非“迁移漏文件”导致全部缺失
- 在 `legacy/beyeah/Beyeah1211`、`Beyeah1206` 中也没有实现前端要求的这些同名 API。
- 说明前后端本身就不是同一契约分支，不是你这次搬迁单点漏拷造成。

### 3. 发现“半迁移残留”
- `CartController`/`CartService`/`CartDao`/`CartItem` 是另一套模型（更像 `beyeah380` 体系），但缺 `CartServiceImpl`、缺对应 mapper/table 接通，属于“死代码残留”。

## 结构与实现质量问题（重点）
### P0（直接导致核心功能失效）
1. 前后端接口契约严重不一致  
- 前端声明：`frontend/src/service/goods.js:11,25`、`detail.js:5`、`cart.js:5,15,26`、`order.js:5,15,25`。  
- 后端不存在对应映射（控制器检索无同名路由）。

2. 购物车备用控制器默认被关闭  
- `backend/src/main/java/SCG/beyeah1211/controller/mall/CartController.java:17` 使用 `@ConditionalOnProperty`，配置里无 `legacy.cart.enabled`，因此不生效。

3. 页面路由与资源链路不完整  
- `backend/pom.xml:30` 的 `spring-boot-starter-thymeleaf` 被注释；资源目录无模板，导致大量 `return "mall/xxx"` 的路由不可用。

### P1（高风险设计/实现缺陷）
1. 返回体协议混乱（两套 Result）  
- `backend/src/main/java/SCG/beyeah1211/util/Result.java`（`resultCode/message`）  
- `backend/src/main/java/SCG/beyeah1211/dao/Result.java`（`code/success/message`）  
- 即便路由接通，也会造成前端解析不一致。

2. 管理端接口缺少统一鉴权拦截  
- 实测 `GET /admin/goods/list`、`/admin/orders/list` 未登录可直接访问数据。  
- 项目中未发现有效拦截器注册。

3. Session 依赖与前后端分离模型冲突  
- 订单/购物车控制器大量直接读取 `HttpSession`（如 `OrderController.java:39`、`ShoppingCartController.java:36`）。  
- 前端 Axios 未配置跨域凭据，且调用的是另一套无 Session API，架构天然冲突。

4. MyBatis Mapper 存在明显错误写法  
- `backend/src/main/java/SCG/beyeah1211/dao/MallUserMapper.java:23` 定义了 `static updateByPrimaryKeySelective`，实际调用会绕过 mapper。

### P2（可维护性与稳定性问题）
1. 数据访问层风格混用  
- 同一用户链路混用 MyBatis + `JdbcTemplate` + 手写 JDBC（`PersonalController`、`MallUser`、`DatabaseUtil`）。

2. 硬编码数据库账号密码  
- `DatabaseUtil.java:10-19`、`application.properties:20-21`。

3. 前端存在“假成功”和状态同步 bug  
- 加购后无论接口结果都弹成功：`frontend/src/views/ProductDetail.vue:37-49`。  
- 删除传参错误：`UserCart.vue:71` 调用 `cartStore.delCart(i)`，store 预期 `goodsId`。  
- 登出未真正清空购物车：`frontend/src/stores/user.js:15` 少了 `()`。  
- 结算总价非响应式：`frontend/src/views/Checkout.vue:53`。

4. 编码与文本问题普遍（中文乱码、注释不可读），影响长期维护。

## 依赖健康度说明（为什么“这次居然没炸”）
1. 前端依赖锁相对稳定  
- `package-lock.json` 存在，`npm run build` 可通过。

2. 后端可编译不代表业务可用  
- 使用 `mvnw` 编译通过，但这是“编译通过”，不是“链路完整”。

3. 隐性风险仍在  
- `pom.xml` 里 `java.version=1.8`，但你当前环境多半是高版本 JDK；短期可跑，长期升级会继续踩兼容坑。  
- MyBatis 与 MyBatis-Plus 混用，历史上最容易触发扫描/绑定问题。

## 建议修复路线（务实版）
### 第一阶段：恢复可交易最小闭环（优先）
1. 选一个统一契约并冻结：建议后端新增“兼容层 API”（沿用前端现有 `/getGoodsByCate` 等），不要先大改前端页面。  
2. 直接弃用当前“残缺 CartController 体系”，基于 `beyeah_mall_shopping_cart_item + beyeahOrderService` 做标准 JSON API。  
3. 统一返回体为一套 `Result`。  
4. 补齐错误码与空指针防护（session/null 检查）。  
5. 修掉前端 4 个关键 bug（上面 P2 第 3 点）。

### 第二阶段：把“能跑”变成“可靠”
1. 管理端鉴权与权限控制（至少 session/token + 拦截器）。  
2. 去掉硬编码密钥，改环境变量。  
3. 建最小自动化回归：登录、商品列表、加购、下单、查订单 5 条集成测试。  
4. 清理乱码与重复/死代码。
