## 2. 当前问题快照（代码锚点）
1. 前端接口契约与后端路由不一致  
- 前端接口定义：`frontend/src/service/goods.js:11,25`、`frontend/src/service/detail.js:5`、`frontend/src/service/cart.js:5,15,26`、`frontend/src/service/order.js:5,15,25`  
- 后端当前是另一套路由：`backend/src/main/java/SCG/beyeah1211/controller/mall/GoodsController.java:42,76`、`ShoppingCartController.java:36,63,78,93`、`OrderController.java:45,60`

2. 代码存在“混合架构 + 死分支”  
- `CartController` 被条件开关关闭：`backend/src/main/java/SCG/beyeah1211/controller/mall/CartController.java:17`  
- 用户链路混用 MyBatis/JdbcTemplate/JDBC：`backend/src/main/java/SCG/beyeah1211/controller/mall/PersonalController.java:125,165,179`、`backend/src/main/java/SCG/beyeah1211/util/DatabaseUtil.java:10-18`、`backend/src/main/java/SCG/beyeah1211/entity/MallUser.java:35-107`

3. 前端存在关键逻辑问题  
- 删除购物车传参错误：`frontend/src/views/UserCart.vue:71`  
- 登出未真正清空购物车：`frontend/src/stores/user.js:15`  
- 结算总价非响应式：`frontend/src/views/Checkout.vue:53`  
- 加购提示“假成功”：`frontend/src/views/ProductDetail.vue:45`

## 3. 本次改造总原则
1. 不做“叠屎山兼容层”，直接统一新契约。  
2. 尽量不大改页面组件，优先改 `frontend/src/service` 和 store。  
3. 后端从“页面路由 + session + 混合数据访问”收敛为“纯 REST JSON + 单一数据访问风格”。  
4. 所有阶段都给出可执行验收项；没有验收通过不进入下一阶段。

## 4. 目标架构（改造后）
1. API 契约  
- 统一前缀：`/api/v1`  
- 统一返回：`ApiResponse<T> { code, message, data, traceId, ts }`  
- 统一错误码（4xx/5xx + 业务码）

2. 后端  
- 只保留 REST 控制器（`@RestController`），不再依赖页面模板路由。  
- 仅保留 MyBatis（去除手写 JDBC 的业务路径）。  
- 认证从 session 迁移到 token（建议 JWT + 刷新策略，或短期 simple token）。

3. 前端  
- 页面组件结构基本不动。  
- `frontend/src/service/*.js` 改为新 API。  
- 补充统一请求错误处理和登录态处理。

## 5. 阶段目标总览
1. 第一阶段（功能闭环）  
- 完成商品列表/搜索/详情、购物车、下单、订单查询闭环。  
- 清除当前残缺路径与死代码，保证“主链路唯一”。

2. 第二阶段（稳定性与工程化）  
- 鉴权、配置、测试、日志、异常、依赖治理到可维护水平。

3. 第三阶段（电商能力增强）  
- 按优先级分批建设：防超卖 > 地址簿 > 退款售后 > 其余增强。

## 6. 里程碑与退出条件
1. M1（阶段一完成）  
- 前端现有页面在不大改 UI 的前提下，核心交易流程可用。  
- 所有核心接口有 Postman/HTTPie 用例并通过。

2. M2（阶段二完成）  
- 无明文数据库密码入库。  
- 有集成测试、最小权限控制、统一日志与异常。

3. M3（阶段三完成）  
- 防超卖、地址簿、退款至少第一批上线并通过回归。
