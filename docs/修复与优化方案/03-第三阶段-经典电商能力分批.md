# 第三阶段：经典电商能力分批建设（按优先级）

## 0. 优先级总览
按你给的优先级落地，并补充依赖关系：
1. 防超卖（最高优先）  
2. 地址簿  
3. 退款/售后  
4. 其他能力（优惠、评价、物流、消息、风控等）

## 1. 批次划分（建议 4 批）
## 批次 A（P0）：防超卖 + 幂等 + 订单超时关闭
### 目标
在并发下保证库存与订单一致，不出现负库存/重复下单。

### 现状锚点
- 下单主流程：`backend/src/main/java/SCG/beyeah1211/service/impl/beyeahOrderServiceImpl.java:183`  
- 扣库存 mapper：`backend/src/main/java/SCG/beyeah1211/dao/beyeahGoodsMapper.java:46`  
- `order_status/pay_status` 字段可复用：`beyeah_mall_order`

### 改造项
1. 下单幂等键（`idempotency_key`）  
- 同一用户重复提交相同请求，只生成一个有效订单。

2. 库存扣减原子化  
- SQL 改为条件更新：`stock_num >= count` 时才扣减。  
- 扣减失败直接返回“库存不足”。

3. 事务边界重构  
- 下单全流程单事务：校验库存 -> 扣库存 -> 写订单 -> 删购物车。  
- 避免“扣了库存但订单失败”。

4. 订单超时关闭任务  
- 新增定时任务关闭超时未支付订单并回补库存。

### 验收
1. 100 并发下单同一商品，库存不为负。  
2. 重复点击支付前提交，订单只产生一次。  
3. 超时订单自动关闭并回补库存。

## 批次 B（P1）：地址簿（你列的第 2 优先）
### 目标
支持用户多地址管理，且订单保留地址快照。

### 改造项
1. 新表 `beyeah_mall_user_address`  
- 字段：`address_id,user_id,consignee,phone,province,city,district,detail,is_default,is_deleted,create_time,update_time`

2. 新接口  
- `GET /api/v1/addresses`  
- `POST /api/v1/addresses`  
- `PUT /api/v1/addresses/{id}`  
- `DELETE /api/v1/addresses/{id}`  
- `PUT /api/v1/addresses/{id}/default`

3. 下单时使用地址 ID，写入订单快照  
- 订单仍保留 `user_name/user_phone/user_address` 快照字段，防止后续地址修改影响历史订单。

### 前端改造范围
1. 仅 `Checkout` 与个人中心地址管理区块。  
2. 商品展示页无须大改。

### 验收
1. 多地址增删改查可用。  
2. 默认地址唯一。  
3. 修改地址不影响历史订单收货信息。

## 批次 C（P1）：退款/售后（你列的第 3 优先）
### 目标
在不打断现有订单履约主链路的前提下，建立闭环售后流程：申请 -> 审核 -> 退款 -> 状态追踪。

### 范围边界（V1）
1. 仅支持已支付订单（`pay_status=1`）的“仅退款”场景，不做退货物流。  
2. 按订单项申请售后；同一 `order_item_id` 只允许 1 条进行中的售后单。  
3. 退款执行先走“站内模拟通道”闭环，预留三方支付回调幂等扩展位。  
4. 管理审核先走 API（`/api/v1/admin/after-sales/**`），不绑定 legacy `/admin` 模板页面。

### 现状锚点（已核对）
- 订单核心流程：`backend/src/main/java/SCG/beyeah1211/service/impl/beyeahOrderServiceImpl.java:191`  
- 现有订单状态枚举：`backend/src/main/java/SCG/beyeah1211/common/beyeahOrderStatusEnum.java:12`  
- 现有支付状态枚举：`backend/src/main/java/SCG/beyeah1211/common/PayStatusEnum.java:12`  
- API 鉴权拦截范围（尚未覆盖售后路由）：`backend/src/main/java/SCG/beyeah1211/config/WebMvcConfig.java:28`  
- 前端订单状态展示入口：`frontend/src/views/UserInfo.vue:207`

### 改造项
1. 数据库与迁移  
- 新增迁移脚本：`database/migrations/V20260308_03_batch_c_after_sale_refund.sql`。  
- 新增表：`beyeah_mall_after_sale`（售后申请）、`beyeah_mall_refund`（退款执行流水）。  
- `beyeah_mall_order` 新增退款聚合字段：`refund_status`、`refund_time`（与 `pay_status` 解耦）。  
- 索引与约束：  
  - `uk_after_sale_order_item(order_item_id)`（V1 严格限制同订单项单次售后）  
  - `uk_refund_after_sale(after_sale_id)`（单售后单仅对应一条退款主流水）  
  - `uk_refund_idempotency(idempotency_key)`（回调幂等防重）

2. 状态机设计（V1）  
- 售后单 `after_sale_status`：`0=待审核, 1=审核通过, -1=审核拒绝, 2=退款中, 3=已退款`。  
- 退款单 `refund_status`：`0=待执行, 1=成功, -1=失败`。  
- 订单退款态 `refund_status`：`0=无退款, 1=退款中, 2=已退款, -1=退款拒绝`。  
- 前端订单展示采用“履约状态 + 退款状态”组合文案，避免覆写现有 `order_status` 语义。

3. 新接口（用户侧 + 管理侧）  
- `POST /api/v1/after-sales`（申请）  
- `GET /api/v1/after-sales`（用户列表）  
- `GET /api/v1/after-sales/{afterSaleNo}`（详情）  
- `GET /api/v1/admin/after-sales`（管理审核）  
- `PUT /api/v1/admin/after-sales/{id}/approve`（审核通过并触发退款执行）  
- `PUT /api/v1/admin/after-sales/{id}/reject`（审核拒绝）

4. 幂等与风控  
- 用户提交售后申请前校验“订单归属 + 支付状态 + 订单项存在性 + 重复申请”。  
- 审核通过动作加事务锁：避免并发审核导致重复退款。  
- 退款执行/回调以 `idempotency_key` 去重，重复请求仅返回已存在结果。

5. 测试与可观测性  
- 集成测试新增：申请成功、重复申请、无权限申请、审核通过、审核拒绝、重复审核幂等。  
- E2E 新增：订单详情页申请售后 -> 用户侧查看状态流转。  
- 日志新增关键字段：`orderNo, afterSaleNo, refundNo, userId, idempotencyKey`。

### 验收
1. 申请、审核、退款状态可追踪。  
2. 重复退款请求不会重复退款。  
3. 订单退款状态与售后/退款流水一致。  
4. 退款链路覆盖成功与失败路径，并具备可回放排障日志。

## 批次 D（P2）：增强能力包（可并行拆分）
1. 促销与优惠券（满减/折扣/券核销）  
2. 评价与追评  
3. 物流轨迹与通知  
4. 收藏/最近浏览/推荐  
5. 风控与审计日志

建议顺序：`优惠券 -> 物流通知 -> 评价 -> 收藏推荐 -> 风控深化`。

## 2. 批次依赖关系
1. A 是 B/C/D 的基础（库存与订单一致性必须先稳定）。  
2. B 与 C 可部分并行，但 C 依赖 A 完成后的订单状态机规范。  
3. D 可穿插执行，但不应阻塞 A/B/C。

## 3. 建议排期（可调）
1. 批次 A：2~3 周  
2. 批次 B：1~2 周  
3. 批次 C：2~3 周  
4. 批次 D：持续迭代

## 4. 每批统一验收模板
1. 数据库迁移脚本可回滚。  
2. OpenAPI 文档更新。  
3. 集成测试覆盖成功与失败路径。  
4. 线上日志可定位每个状态变迁。
