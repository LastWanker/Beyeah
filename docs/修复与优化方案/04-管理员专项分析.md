# 第四阶段：管理员专项分析（RBAC + 能力串联）

> 文档时间：2026-03-08  
> 目标：回答三个问题  
> 1) 管理员能力是否真实可用  
> 2) RBAC 是否缺失，以及如何补齐  
> 3) 前端与数据库是否具备管理后台演进基础

## 0. 结论摘要
1. 管理员“业务能力代码”不是空壳，商品/订单/用户/分类/轮播/首页配置等管理接口在后端已有实现。  
2. 但“权限与链路”不完整：当前没有真正的 RBAC；`/admin/**` 缺少统一鉴权拦截；`/api/v1/admin/after-sales/**` 仅要求登录态，不校验管理员角色。  
3. 前端没有独立管理员控制台（现有前端路由仅面向 C 端），导致管理功能难以稳定可视化落地。  
4. 数据库仅有 `beyeah_mall_admin_user` 基础账号表，没有角色、权限、用户角色、角色权限、操作审计等核心表。  
5. 现有管理员功能“可调用但不可靠”，风险主要在越权、可追溯性、回归覆盖和线上治理缺失。

## 1. 现状盘点（代码事实）
## 1.1 已存在的管理员功能（后端）
1. 管理员登录与资料：`backend/src/main/java/SCG/beyeah1211/controller/admin/AdminController.java`  
2. 商品管理：`backend/src/main/java/SCG/beyeah1211/controller/admin/beyeahGoodsController.java`  
3. 订单管理：`backend/src/main/java/SCG/beyeah1211/controller/admin/beyeahOrderController.java`  
4. 用户封禁/解封：`backend/src/main/java/SCG/beyeah1211/controller/admin/beyeahUserController.java`  
5. 分类/轮播/首页配置：`controller/admin` 下相关控制器  
6. 管理侧售后审核 API：`backend/src/main/java/SCG/beyeah1211/controller/api/AdminAfterSaleApiController.java`

## 1.2 权限模型现状
1. 仅存在 API Token 鉴权拦截：`backend/src/main/java/SCG/beyeah1211/config/WebMvcConfig.java`。  
2. 拦截范围包含 `/api/v1/admin/after-sales/**`，但不区分“普通用户”与“管理员”。  
3. `/admin/**` 传统后台路由未看到统一拦截器接入。  
4. 管理员登录依赖 session，但权限粒度仅“登录成功/失败”。

## 1.3 前端现状
1. 当前前端路由仅有 `/login / /info /cart /checkout /product /category /search`。  
2. 无 `/admin` 前端页面与菜单体系。  
3. 说明管理员业务无法通过当前 C 端前端完成标准化操作闭环。

## 1.4 数据库现状
1. 存在管理员账号表：`beyeah_mall_admin_user`。  
2. 字段仅包含 `login_user_name/login_password/nick_name/locked`。  
3. 缺少 RBAC 核心结构：角色表、权限表、用户-角色关系、角色-权限关系、操作审计表。

## 2. 风险评估（是否“真的可靠”）
## 2.1 高风险
1. 越权风险：管理接口未做角色级授权，存在“登录即可调管理能力”的潜在问题。  
2. 审计风险：缺少操作审计日志，无法追踪“谁在何时修改了商品/订单/用户状态”。  
3. 回归风险：当前测试覆盖以 C 端 API 为主，管理员功能缺少系统化自动化回归。

## 2.2 中风险
1. 一致性风险：订单履约类操作（配货/出库/关闭）有状态校验，但缺少管理员操作幂等防重与并发冲突治理。  
2. 可观测性风险：缺少面向后台操作的指标体系（封禁次数、审核通过率、异常失败率）。

## 2.3 低风险
1. 基础能力风险较低：管理员账号登录、商品/订单/用户基础 CRUD 调用链路存在，可作为重构基础。

## 3. RBAC 目标设计（建议）
## 3.1 角色模型（最小可用）
1. `SUPER_ADMIN`：全量权限。  
2. `MERCHANDISER`：商品、分类、首页配置、轮播。  
3. `WAREHOUSE`：订单配货、出库。  
4. `CUSTOMER_SERVICE`：售后审核、退款处理。  
5. `RISK_OPERATOR`：用户封禁/解封、风控处理。  
6. `AUDITOR`：只读查询（无写权限）。

## 3.2 权限粒度
1. 采用 `resource:action`，示例：`goods:read`, `goods:write`, `order:checkout`, `user:lock`, `after_sale:approve`。  
2. 接口与权限映射在后端集中配置，禁止分散硬编码。

## 3.3 鉴权链路
1. 新增 `AdminAuthInterceptor`：校验 admin token/session，注入 admin 上下文。  
2. 新增 `@RequirePermission` 注解 + 权限拦截器。  
3. `/api/v1/admin/**` 全量强制走管理员鉴权链路，不复用 C 端用户鉴权。

## 4. 管理员功能串联（从“能调接口”到“可运营”）
## 4.1 商品管理链路
1. 登录管理员 -> 商品列表 -> 新增/编辑 -> 上下架 -> 审计落库。  
2. 必备保障：权限校验、字段校验、操作日志、失败告警。

## 4.2 订单履约链路
1. 订单列表 -> 批量配货 -> 出库 -> 异常关闭。  
2. 必备保障：状态机校验、重复操作幂等、操作审计、告警。

## 4.3 用户治理链路
1. 用户列表 -> 封禁/解封 -> 原因记录。  
2. 必备保障：权限最小化、封禁原因必填、可追溯。

## 4.4 售后审核链路
1. 售后申请列表 -> 审核通过/拒绝 -> 退款状态跟踪。  
2. 必备保障：角色限定（客服/管理员）、审核意见留痕、幂等防重。

## 5. 数据库补齐方案（RBAC + 审计）
## 5.1 新增表（建议）
1. `beyeah_admin_role`（角色主数据）  
2. `beyeah_admin_permission`（权限主数据）  
3. `beyeah_admin_user_role`（管理员-角色）  
4. `beyeah_admin_role_permission`（角色-权限）  
5. `beyeah_admin_operation_log`（操作审计日志）

## 5.2 迁移策略
1. 新增迁移脚本：`database/migrations/V202603xx_04_admin_rbac.sql`  
2. 默认创建 `SUPER_ADMIN` 角色并绑定全权限。  
3. 将历史 admin 账号回填到角色关系表。  
4. 迁移脚本必须可重复执行、可回滚。

## 6. 前端补齐方案（管理员控制台）
## 6.1 建议路线
1. 新建独立后台前端（建议 `frontend-admin/`）避免污染 C 端路由。  
2. 或在现有前端新增 `/admin/*` 路由分区与独立布局。  
3. 管理员登录与 C 端登录 token 分离存储，避免混用。

## 6.2 最小页面集
1. 登录页  
2. 仪表盘  
3. 商品管理页  
4. 订单履约页  
5. 用户治理页  
6. 售后审核页  
7. 审计日志页

## 7. 可靠性改造清单（与 RBAC 并行）
1. 管理员接口自动化测试：登录、鉴权拒绝、权限拒绝、关键写操作成功/失败路径。  
2. 关键操作审计：记录操作者、对象、前后值、请求 ID、IP。  
3. 幂等与并发：对批量状态流转接口加防重键与状态版本校验。  
4. 观测指标：管理操作成功率、失败率、拒绝率、平均耗时、异常分布。  
5. 安全基线：密码策略、登录失败限流、敏感接口二次确认。

## 8. 建议排期（激进版）
1. 第 1 周：RBAC 表结构 + 管理员鉴权链路 + 权限注解。  
2. 第 2 周：商品/订单/用户/售后管理接口权限化改造 + 审计日志。  
3. 第 3 周：管理员前端最小可用页面 + 联调 + 自动化回归。  
4. 第 4 周：灰度上线与告警收敛。

## 9. 验收标准（必须全部满足）
1. `/api/v1/admin/**` 未登录返回 401，非管理员返回 403。  
2. 管理员角色权限差异可验证（同接口不同角色结果不同）。  
3. 商品、订单、用户、售后四条链路至少各 1 条完整 E2E。  
4. 所有管理写操作均有审计日志，可按 `traceId` 回放。  
5. 安全扫描与回归测试通过后方可上线。
